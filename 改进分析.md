## 改进余地较大代码案例分析

---

### 问题一：GameSession 接口违反接口隔离原则（ISP）

#### 问题描述

`GameSession` 将四种玩法差异巨大的游戏强行统一在一个接口中，各实现类被迫编写无意义的方法：

```java
public interface GameSession {
    Map<Disc, Integer> counts();          // Minesweeper 永远返回空 Map
    Set<Position> validMoves(Disc disc);  // Chess/Minesweeper 忽略参数
    Disc current();                       // Minesweeper 永远返回 BLACK
    ActionResult tryPass();               // 仅 Reversi 需要

    default ActionResult tryFlag(Position p) {
        return ActionResult.INVALID_MOVE;  // 只有扫雷需要，其余靠默认方法蒙混
    }
}
```

| 方法 | Minesweeper | Chess | Peace |
|------|:-----------:|:-----:|:-----:|
| `counts()` | 空实现 | 语义不匹配 | ✓ |
| `validMoves(Disc)` | 忽略参数 | 空实现 | ✓ |
| `tryPass()` | 不支持 | 不支持 | 不支持 |
| `board()` | 仅取尺寸（自有 `char[][]`） | 仅取尺寸（自有 `char[][]`） | ✓ |

#### 解决方案

```java
// 所有游戏共有
public interface GameSession {
    GameMode mode();
    boolean isQuit();   void quit();
    boolean isOver();
    String cellDisplay(Position p);   // 消除 UI 层 instanceof 分支
    ActionResult tryMove(Position p);
}

// 棋盘类（Peace、Reversi）
public interface BoardGameSession extends GameSession {
    Board board();  Disc current();
    Map<Disc, Integer> counts();
    Set<Position> validMoves();
    ActionResult tryPass();
}

// 国际象棋
public interface ChessSession extends GameSession {
    char pieceAt(Position p);
    ActionResult tryMove(Position from, Position to, Character promotion);
    String lastMessage();   Disc winner();   int moveCount();
}

// 扫雷
public interface MinesweeperSession extends GameSession {
    ActionResult tryFlag(Position p);
    int mineCount();   int flagCount();   String lastMessage();
}
```

#### 改进理由

1. **符合 ISP**：各实现类仅依赖真正需要的方法，消除空实现。
2. **消除 `instanceof` 分支**：UI 渲染从 `if (game.mode() == ...)` 变为 `game.cellDisplay(p)`，遵循"Tell, Don't Ask"。
3. **新增游戏零波及**：如加入 Go，仅需实现 `BoardGameSession`，不影响 Chess/Minesweeper 现有代码。

### 问题二：插件注册硬编码，无法做到真正的即插即用

#### 问题描述

项目虽引入 `GamePlugin` 接口，但注册仍硬编码在源码中。新增一个游戏（如 Go）需同时改动 **3 处**：

```java
// 1. GameMode.java — 枚举硬编码
public enum GameMode { PEACE, REVERSI, MINESWEEPER, CHESS }  // 必须加 GO

// 2. GameRegistry.java — 列表硬编码
public static GameRegistry defaultRegistry() {
    return new GameRegistry(List.of(new PeaceGamePlugin(), ...));  // 必须加一行
}

// 3. GoGamePlugin.java — 新建但无自动发现
```

这违背插件化初衷——**真正插件化应做到新增游戏不修改现有代码**。

#### 解决方案

**Step 1：去枚举化 —— `GameMode` 改为值对象**

```java
public record GameMode(String id) {
    public static GameMode of(String id) {
        return new GameMode(id.toLowerCase(Locale.ROOT));
    }
}
```

枚举是编译期封闭集合，值对象支持 `new GameMode("go")` 即可，无需动源码。

**Step 2：Java SPI 自动发现插件**

```java
public final class GameRegistry {
    public static GameRegistry autoDiscover() {
        ServiceLoader<GamePlugin> loader = ServiceLoader.load(GamePlugin.class);
        List<GamePlugin> plugins = new ArrayList<>();
        loader.forEach(plugins::add);
        return new GameRegistry(plugins);
    }
}
```

**Step 3：各插件在 `META-INF/services/` 声明自己**

```
# META-INF/services/reversi.gamehall.GamePlugin
reversi.games.chess.ChessGamePlugin
reversi.games.reversi.ReversiGamePlugin
```

**效果**：新增游戏只需实现接口 + 声明一行配置 + 放入 classpath，**无需修改平台层任何源码**。

#### 改进理由

1. **符合开闭原则（OCP）**：对扩展开放，对修改关闭。
2. **独立部署**：不同游戏可由不同开发者独立打包 JAR，互不干扰。
3. **消除枚举瓶颈**：编译期封闭集合与灵活扩展天然冲突，字符串 ID 支持任意扩展。
4. **业界标准**：Java SPI 是 JDK 内置机制（JDBC 驱动、SLF4J 均采用），零第三方依赖。
5. **降级兼容**：若 SPI 过重，可退用 `plugins.json` 配置文件驱动，同样避免修改源码。

---

### 总结

| 问题 | 违反原则 | 核心改动 | 影响范围 |
|------|----------|----------|----------|
| 胖接口 | 接口隔离原则 (ISP) | 拆为 4 个子接口 | `core/` 包 4 个实现类 |
| 硬编码插件 | 开闭原则 (OCP) | SPI 自动发现 + 去枚举化 | `GameRegistry` + `GameMode` |

二者协同：**ISP 拆分**让新游戏只需实现最少契约；**SPI 机制**让新游戏无需触碰平台代码，将当前"改 3 处源码才能加游戏"变为"打包一个独立 JAR 即可"。
